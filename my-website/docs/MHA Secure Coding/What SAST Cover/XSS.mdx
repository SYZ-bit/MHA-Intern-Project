# Cross-Site Scripting

Cross-Site Scripting (also known as XSS) attacks are a type of injection, in which malicious scripts are injected into trusted websites.
XSS attacks occurs when an attacker send malicious code into the website, which allows them to execute code in the victim's browser, potentially stealing data or hijacking user sessions.
Flaws that allow these attacks are quite widespread and any web application that uses input from a user can have these flaws if the output it generates is not validated or encoded.

When an attacker use XSS to send a malicious script to unsuspecting user, the end user's browser has no way to know what the script should not be trusted, and will execute the malicious script. 
The end user's browser will think the script came from a trusted source, the malicious script can access any cookies, session tokens, or other sensitive information retained by the browser and used with that site.
These script can even rewrite the content of the HTML page.

## Types of XSS

### Reflected XSS

Reflected XSS attacks arise when a web server reflects an injected script, such as a search result, an error message, or any other response that includes some or all of the input sent to the server as part of the request.

The attack is then delivered to the victim through another route (e.g., e-mail or an alternative website), thus tricking the user into clicking on a malicious link. The injected code travels to the vulnerable website, which reflects the attack payload back to the user's browser. The browser then executes the code because it came from a "trusted" server.

### Stored XSS

In the Stored XSS attack, the injected script is stored on the target application as legitimate content, such as a message in a forum or a comment in a blog post. The injected code is stored in the database and sent to the users when it is retrieved, thus executing the attack payload in the victim's browser.

### DOM-based XSS

DOM-based XSS vulnerabilities typically arise when JavaScript on a webpage takes user-supplied input from a source like document.location and passes it to a function that can execute code, such as innerHTML. In a typical attack, the malicious payload is delivered to the victim via an indirect method—such as an email or a different website—tricking them into clicking a harmful link. The attack is executed entirely on the client side, with the malicious code running instantly in the user’s browser.

## Impact

Cross site scripting attacks can have devastating consequences. Code injected into a vulnerable application can exfiltrate data or install malware on the user’s machine. Attackers can masquerade as authorized users via session cookies, allowing them to perform any action allowed by the user account.

XSS can also impact a business’s reputation. An attacker can deface a corporate website by altering its content, thereby damaging the company’s image or spreading misinformation. A hacker can also change the instructions given to users who visit the target website, misdirecting their behavior. This scenario is particularly dangerous if the target is a government website or provides vital resources in times of crisis.

## Prevention

XSS attacks can be mitigated by performing appropriate server-side validation and escaping. Remediation relies on performing Output Encoding (e.g., using an escape syntax) for the type of HTML context into which untrusted data is reflected.

### Input Validation

- Exact Match: Only accept values from a finite list of known values.
- Allow list: If a list of all the possible values can't be created, accept only known good data and reject all unexpected input.
- Deny list: If an allow-list approach is not feasible (on free-form text areas, for example), reject all known bad values.

### Output Encoding

Output encoding is a technique used to transform untrusted input into a safe format so that it is displayed as data in the browser rather than executed as code. This encoding is applied when data is sent from the application to a downstream component. The table below outlines various downstream contexts where untrusted input might appear, along with the appropriate encoding method:

| Context         | Code Example                                                                                 | Encoding Method           |
|----------------|----------------------------------------------------------------------------------------------|---------------------------|
| HTML Body       | `<div>USER-CONTROLLED-DATA</div>`                                                            | HTML Encoding             |
| HTML Attribute  | `<input type="text" value="USER-CONTROLLED-DATA" />`                                         | HTML Attribute Encoding   |
| URL Parameter   | `<a href="/search?value=USER-CONTROLLED-DATA">Search</a>`                                    | URL Encoding              |
| CSS             | `<div style="width: USER-CONTROLLED-DATA;">Selection</div>`                                  | CSS Hex Encoding          |
| JavaScript      | `<script>var lang = 'USER-CONTROLLED-DATA';</script>, <script>setLanguage('USER-CONTROLLED-DATA');</script>` | JavaScript Encoding       |

### Defense in Depth

#### Content Security Policy (CSP)
Content Security Policy (CSP) is a browser-level security feature that allows web applications to define a whitelist of trusted sources for loading client-side resources such as JavaScript, CSS, and images. By using a specific HTTP header, CSP tells the browser to only execute or render content from those explicitly allowed sources.

```
Content-Security-Policy: default-src: 'self'; script-src: 'self' static.domain.tld
```

The above policy instructs the browser to load all resources from the site's own origin ('self') and only allow JavaScript files from the same origin and static.domain.tld. For more information on configuring CSP, refer to the official documentation or a detailed guide.

#### Content Types
To avoid browsers misinterpreting non-HTML HTTP responses as executable HTML or JavaScript, it's important to always include a correct Content-Type header in the HTTP response. This ensures the browser handles the content as intended, reducing the risk of content-type-based attacks.

#### Modern Frameworks
Modern JavaScript frameworks such as Angular and React, as well as server-side templating engines like Go Templates, come with built-in mechanisms to prevent Reflected Cross-Site Scripting (XSS). Leveraging these frameworks properly can significantly reduce the risk of introducing XSS vulnerabilities.
