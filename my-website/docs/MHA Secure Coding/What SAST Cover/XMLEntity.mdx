# XML Entity Expansion

## Description

XML entities are a way of representing an item of data within an XML document, instead of using the data itself. Various entities are built in to the specification of the XML language. For example, the entities &lt; and &gt; represent the characters < and >. These are metacharacters used to denote XML tags, and so must generally be represented using their entities when they appear within data.

XML External Entity (XXE) attacks target applications that process XML input by exploiting the parser’s ability to handle external entities. By submitting malicious XML data that defines external entities, attackers can take advantage of insecure parser configurations to access files on the server, perform denial-of-service (DoS) attacks, or communicate with internal systems the application can reach.

These vulnerabilities stem from the XML format’s built-in features, some of which can be dangerous if not properly managed. Given the potential impact and continued occurrence of XXE attacks, they are included in the OWASP Top 10 list under the Security Misconfiguration category.

As with many other security issues, the frequency of XXE vulnerabilities could be significantly reduced through regular, up-to-date secure coding training for developers. 

## Impact 

XXE attacks can lead to denial-of-service (DoS) incidents and the exposure of sensitive local files, such as passwords or private user data. Since the attack targets the application processing the XML, it can also be used to move laterally within a network, potentially enabling Server-Side Request Forgery (SSRF) attacks against internal systems that lack proper protections.

Despite XML-related vulnerabilities being known for nearly two decades, major tech companies like Google and Facebook have still encountered such issues in recent years. This highlights the ongoing risk that misconfigurations and insecure coding practices can pose—often resulting in significant consequences, including regulatory fines.

## Scenarios

To understand XXE vulnerabilities, it's important to focus on two key features of XML: XML entities and the Document Type Definition (DTD).

XML entities allow data within an XML document to be represented using placeholders instead of actual content. For instance, predefined entities like &lt; and &gt; represent the < and > characters. Developers can define custom entities using DTDs.

Document Type Definitions (DTDs) define the structure and allowed elements of an XML document and are often used for validation. A DTD can be embedded directly in an XML document using a DOCTYPE declaration, or it can reference an external source via a URL.

### Denial-of-Service (DoS) Attacks

Malicious actors can exploit XML entities to overload the XML parser and cause denial-of-service conditions. One common technique is nesting entities within each other—leading to massive memory consumption. A well-known example is the Billion Laughs attack, where entities recursively expand into large blocks of text, overwhelming the parser.

```
POST http://www.vulnerableapp.com/xml HTTP/1.1

<?xml version="1.0" encoding="ISO-8859-1"?> 
<!DOCTYPE foo [
  <!ELEMENT foo ANY>
  <!ENTITY bar "MHA ">
  <!ENTITY t1 "&bar;&bar;">
  <!ENTITY t2 "&t1;&t1;&t1;&t1;">
  <!ENTITY t3 "&t2;&t2;&t2;&t2;&t2;">
]>
<foo>
  Join &t3;
</foo>
```
```
HTTP/1.0 200 OK

Join MHA MHA MHA MHA MHA MHA MHA MHA MHA MHA MHA MHA MHA MHA MHA MHA MHA MHA MHA MHA MHA MHA MHA MHA MHA MHA MHA MHA MHA MHA MHA MHA MHA MHA MHA MHA (repeated many times)
```

This recursive expansion can exhaust system resources, crashing or slowing down the application.



### Server-Side Request Forgery Attacks

XML external entities can also be used to retrieve local files from the system, such as /etc/passwd, which may contain sensitive information.

**Request**
```
POST http://www.vulnerableapp.com/xml HTTP/1.1

<?xml version="1.0" encoding="ISO-8859-1"?> 
<!DOCTYPE foo [
  <!ELEMENT foo ANY>
  <!ENTITY xxe SYSTEM
  "file:///etc/passwd">
]>
<foo>
  &xxe;
</foo>
```
 
**Response**

```
HTTP/1.0 200 OK
 
root:x:0:0:root:/root:/bin/bash
daemon:x:1:1:daemon:/usr/sbin:/bin/sh 
bin:x:2:2:bin:/bin:/bin/sh
sys:x:3:3:sys:/dev:/bin/sh 
(...)
```

Attackers can also direct the entity to a URL, effectively tricking the server into making HTTP requests to internal systems—this is known as Server-Side Request Forgery (SSRF).

**Request**

```
POST http://www.vulnerableapp.com/xml HTTP/1.1

<?xml version="1.0" encoding="ISO-8859-1"?> 
<!DOCTYPE foo [
  <!ELEMENT foo ANY>
  <!ENTITY xxe SYSTEM
  "http://internal.vulnerableapp.com:8443">
]>
<foo>
  &xxe;
</foo>
```

**Response**

```
HTTP/1.0 200 OK
 
(.. result of the request to http://internal.vulnerableapp.com:8443 ...)
```

By exploiting SSRF, attackers can pivot to internal or backend systems that are otherwise inaccessible from the outside.

## Prevention

Disabling support for Document Type Definitions (DTDs) is one of the most effective ways to prevent XXE attacks.

Whenever feasible, it's best to use simpler data formats like JSON. For nearly a decade, JSON has been preferred over XML due to its lightweight structure and more modern design.

However, exceptions do exist. In scenarios where disabling DTDs is not an option due to business requirements, and switching to a different format isn’t viable, developers must implement additional safeguards.

If the entire XML document comes from an untrusted source, it’s typically not possible to sanitize or validate potentially malicious input embedded in the DTD's system identifiers. In such cases, the XML processor should be explicitly configured to use a local, static DTD and block any external or embedded DTDs declared within incoming XML documents.

## Python example

Python includes a built-in XML parser (xml) that is suitable for basic XML parsing and manipulation. However, it lacks support for advanced features such as XML validation. Although it does not support external entities—reducing the risk of XXE attacks—it may still be susceptible to other XML-related vulnerabilities.

For more advanced XML processing needs, developers often use third-party libraries like lxml. Unlike the native parser, lxml has external entity parsing enabled by default, which can introduce security risks if not properly managed.

### Vulnerable example using lxml

```
@tools.route("/is_xml", methods=['POST'])
def tools_is_xml():
    try:
        # read data from POST
        xml_raw = request.files['xml'].read()

        # create the XML parser
        parser = etree.XMLParser()

        # parse the XML data
        root = etree.fromstring(xml_raw, parser)

        # return a string representation
        xml = etree.tostring(root, pretty_print=True, encoding='unicode')
        return jsonify({'status': 'yes', 'data': xml})
    except Exception as e:
        return jsonify({'status': 'no', 'message': str(e)})
```
When the etree.fromstring method is invoked, it parses and expands the external entities. For example, when uploading the following XML document:

```
<!DOCTYPE d [<!ENTITY e SYSTEM "file:///etc/passwd">]><t>&e;</t>
```

The entity &e; is expanded with the contents of the local /etc/passwd file, resulting in the disclosure of the file's contents.

This vulnerability can be avoided by setting the resolve_entities=False argument when creating the etree.XMLParser. If only simple XML data processing is required, use the native parser.
